# translate — 사이트 한국어 문서화 커맨드

사이트 식별자를 인자로 받아, 미번역·원문 변경 페이지의 원문을 **사람이 읽기 좋은 한국어 기술 문서로
재구성·번역**하고 출력 경로에 저장한다. 원문이 OpenAPI 스펙 등 기계친화 포맷이면 그대로 옮기지 않고
표·설명 등 가독성 있는 구조로 가공한다(SPEC §5.4, §5.6 / ANALYSIS D11).

## 사용법

```
/translate <siteID>
```

예: `/translate ollama`

## 전제 조건

- 해당 사이트의 수집(`go run ./cmd/collect <siteID>`)이 완료되어 원문이 존재해야 한다.
- 사이트 폴더 `sites/<siteID>/`가 존재하고, 그 안에 `config.json`이 있어야 한다.
- 용어집 파일 `sites/<siteID>/glossary.json`이 있어야 한다(없으면 빈 매핑으로 처리).

## 번역 절차

### 1단계: 설정·용어집·번역 대상 확인

1. `sites/<siteID>/config.json`을 읽어 사이트 설정을 로드한다. 경로 규칙은 모두 사이트 폴더 규약으로
   고정된다(출력: `sites/<siteID>/output/`, 용어집: `sites/<siteID>/glossary.json`,
   매니페스트: `sites/<siteID>/manifest.json`).
2. `sites/<siteID>/glossary.json`에서 용어집(원어→한국어 JSON 매핑)을 읽는다.
   파일이 없으면 빈 매핑으로 진행한다.
3. CLI로 번역 대상 목록을 확인한다:
   ```
   go run ./cmd/translate plan <siteID>
   ```
   출력된 목록에서 각 항목의 `PageURL`, 로컬 경로, 원문 해시 앞부분을 확인한다.
4. 대상이 없으면 "번역할 페이지가 없습니다 (모두 최신 번역 상태)"를 출력하고 종료한다.

### 2단계: 각 대상 페이지를 읽기 좋은 한국어 문서로 재구성

`[]TranslationTarget`을 순서대로 처리한다. 각 페이지에 대해 `target.LocalPath`의 원문을 읽고,
아래 기준으로 **사람이 읽기 좋은 한국어 문서**를 만든다.

공통 원칙:
- **목표는 충실한 번역이 아니라 읽기 좋은 문서다.** 원문을 그대로 옮기지 말고, 내용을 이해해 한국어
  기술 문서 독자가 바로 이해할 수 있는 구조로 재구성한다.
- **용어집 우선 적용**: 용어집에 정의된 원어는 반드시 그 번역어로 일관 반영한다.
- **번역문 단독**: 원문을 병기하지 않는다(D1). 문서 끝에 `> 원문: <PageURL>` 한 줄만 출처로 남긴다.
- **보일러플레이트 제거**: 모든 페이지에 반복되는 안내문(예: 상단의 "Documentation Index / Fetch the
  complete documentation index at..." blockquote)은 생략한다.

원문이 OpenAPI 스펙 등 기계친화 포맷(yaml/json 덩어리)인 경우:
- 스펙을 그대로 코드블록으로 두지 않는다. 내용을 파악해 다음과 같이 재구성한다.
  - 제목 + 엔드포인트(예: `POST /api/chat`) + 한국어 한 줄 설명
  - 요청 파라미터·중첩 객체·옵션·응답 필드를 각각 **표**(필드 / 타입 / 필수 / 설명)로 정리
  - `summary`·`description` 등 자연어 설명은 한국어로 번역해 표의 설명 칸·문단에 반영
- **원문 유지 대상**(번역·변형 금지): 필드 키 이름(`model`, `messages` 등), 타입·enum 값, 엔드포인트
  경로, 예시 요청/응답(curl·JSON)·base64 이미지 등 코드. 이들은 코드블록 안에 원문 그대로 둔다.
- 응답 예시·사용 예시(curl 등)는 코드블록으로 보존하되, 각 예시에 한국어 제목/설명을 붙인다.

원문이 산문 위주 마크다운인 경우:
- 자연스러운 한국어 기술 문서로 번역한다. 헤딩·목록·표·링크 등 마크다운 구조는 가독성에 도움이 되는
  선에서 유지하되, 어색하면 한국어 문서 관례에 맞게 다듬는다.
- 코드 블록(실행 명령, 코드 샘플, 설정 예시)은 번역하지 않고 원문 그대로 둔다.

### 3단계: 저장과 매니페스트 기록

각 페이지의 재구성 결과를 경로 규칙에 맞춰 `sites/<siteID>/output/<URL경로구조>.md`에 저장한다.
파일을 모두 저장한 뒤 CLI로 매니페스트에 완료 해시를 일괄 기록한다:

```
go run ./cmd/translate commit <siteID>
```

이 명령이 `sites/<siteID>/output/`에 파일이 존재하는 페이지를 찾아
`sites/<siteID>/manifest.json`의 `translated_hash`를 현재 원문 해시로 갱신한다.
번역문 파일을 작성하지 않은 페이지는 건너뛴다.

끝나면 처리 건수와 출력 경로(`sites/<siteID>/output/`)를 보고한다. 오류가 있으면 해당 페이지 URL과
오류 내용을 함께 보고한다.

## 경로 규칙

| 구분 | 경로 형태 |
|---|---|
| 사이트 폴더 | `sites/<siteID>/` |
| 설정 | `sites/<siteID>/config.json` |
| 용어집 | `sites/<siteID>/glossary.json` |
| 원문 | `sites/<siteID>/raw/<URL경로>.md` |
| 번역문 출력 | `sites/<siteID>/output/<URL경로>.md` |
| 매니페스트 | `sites/<siteID>/manifest.json` |

원문 경로와 번역문 경로는 루트(`raw/` vs `output/`)만 다르고 하위 구조가 동일하다. 즉 파일 단위의
디렉터리 구조는 보존되며, 재구성으로 달라지는 것은 **파일 내부의 구성**이다.

## 증분 동작

이미 번역되어 있고 원문이 바뀌지 않은 페이지는 대상에서 제외된다(매니페스트의 `translated_hash`와
현재 원문 해시 비교 — D8). 재실행하면 미번역분·원문 변경분만 처리된다(SPEC §5.5).

## 비고

- 재구성·번역 텍스트 생성은 이 커맨드(Claude 절차) 소관이다. Go(`internal/translator`)는 대상 선별·
  저장·매니페스트 기록 같은 결정적 부분만 담당하며, 재구성 방식 변경은 Go 코드에 영향을 주지 않는다.
- 사이트 폴더 경로 기본값은 `./sites/<siteID>`다. 실제 경로가 다르면 번역 전에 사용자에게 확인한다.
- 재구성은 비결정적이다(같은 원문도 실행마다 표현이 달라질 수 있다). 증분 판정은 원문 해시 기준이므로,
  원문이 바뀌지 않으면 재번역하지 않는다.
