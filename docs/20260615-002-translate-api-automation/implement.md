- [ ] task-002: 사이트 설정에 모델 선택 필드 추가
  - 목적: 사이트 설정에 호출 모델을 선언할 수 있고, 선언하지 않은 기존 설정도 그대로 유효하게 로드된다.
  - 접근: 사이트 설정 구조체에 선택 모델 필드(JSON `"model"`)를 추가하되 필수 검증 대상에 넣지 않아,
    값이 없으면 빈 값으로 로드되도록 둔다.
  - 검증 조건:
    - 결과: `"model"`이 있는 설정은 그 값이 채워지고, 없는 설정은 빈 값으로 로드되며 기존 필수 검증·동작이
      그대로 통과한다.
    - 확인: 모델 지정 설정·미지정 설정 각각의 로더 단위 테스트가 통과한다(미지정 설정 하위호환 포함).
  - 참조: SPEC §5.4 / ANALYSIS §3, §5 D-c

- [ ] task-003: 재구성 규칙·스타일 프롬프트 자산
  - 목적: 페이지·사이트 불변의 재구성 규칙과 스타일 기준이 프롬프트로 쓸 수 있는 안정된 텍스트로 보유된다.
  - 접근: 재구성 규칙·스타일 기준을 코드 내 상수 텍스트로 보유하되, 내용 기준은 기존 translate 절차와
    일치시킨다(원문 병기 없음·상단 보일러플레이트 제거·문서 끝 출처 한 줄·기계친화 포맷 표·설명 재구성).
  - 검증 조건:
    - 결과: 규칙·스타일 텍스트가 호출마다 동일한 바이트로 제공되고(타임스탬프·비결정 직렬화 없음),
      위 내용 기준 항목이 모두 텍스트에 포함된다.
    - 확인: 자산 텍스트가 위 기준 항목을 담고 있고 호출 간 동일함을 단위 테스트 또는 코드 검토로 확인한다.
  - 참조: SPEC §5.6 / ANALYSIS §1, §5 D-b

- [ ] task-004: 공유 프리픽스 조립과 캐시 컨트롤
  - 목적: 재구성 규칙·스타일(안정부)과 사이트 용어집(가변부)이 하나의 공유 프리픽스로 합쳐지고, 그 프리픽스가
    캐시 대상으로 표시되어 페이지마다 재사용된다.
  - 접근: 규칙·스타일 자산과 사이트별로 로드한 용어집을 안정부·가변부 순서로 합류해 시스템 프리픽스를 만들고,
    그 마지막 블록에 캐시 컨트롤 마커를 항상 부착한다. 용어집은 기존 용어집 로더로 합류한다.
  - 검증 조건:
    - 결과: 합쳐진 프리픽스가 규칙·스타일 다음에 용어집 매핑을 담고, 캐시 마커가 부착되며, 용어집이 비어도
      정상 구성된다(프리픽스가 최소 토큰 미만이면 캐시 없이 동작함을 전제).
    - 확인: 용어집 유무 두 경우에 대해 프리픽스 조립 결과와 캐시 마커 부착을 단위 테스트로 확인한다.
  - 참조: SPEC §5.5, §5.6 / ANALYSIS §2, §5 D-b, D-f

- [ ] task-005: 모델 선택과 API 키 확인
  - 목적: 사이트 설정 모델이 비어 있으면 Sonnet 4.6로, 지정되면 그 값으로 호출 모델이 결정되고, API 키가
    없으면 호출 전에 명확한 오류로 중단한다.
  - 접근: 설정 모델 값이 비었을 때 기본 모델 ID로 대체하는 결정 로직을 두고, 환경변수 API 키 부재를
    클라이언트 구성 시점에 감지해 명확한 오류로 반환(조용히 실패 금지)한다.
  - 검증 조건:
    - 결과: 모델 미지정은 기본 모델로, 지정은 그 값으로 결정되고, 키가 없으면 명확한 오류로 중단(종료 코드 1)된다.
    - 확인: 모델 미지정·지정 두 경우의 모델 결정과 키 부재 시 오류 중단을 단위 테스트로 확인한다.
  - 참조: SPEC §5.4, §5.7 / ANALYSIS §2, §5 D-c

## Section: 동기 경로

- [ ] task-006: 동기 단건 재구성 호출(스트리밍·캐싱·모델 반영)
  - 목적: 원문 한 페이지를 공유 프리픽스와 함께 동기 호출로 재구성해, 잘림 없이 전체 재구성 텍스트와 함께
    응답 모델·캐시 읽기 토큰 같은 관찰 정보를 얻는다.
  - 접근: 공유 프리픽스를 시스템으로, 페이지 원문을 사용자 메시지로 두고 스트리밍 호출로 최종 메시지를
    누적하며 출력 한도를 충분히 크게 둔다. 응답의 보고 모델과 캐시 읽기 토큰을 노출한다. 일시 오류·rate
    limit 재시도는 SDK 기본 동작에 맡기되 필요 시 기존 fetcher 백오프 관례를 참고한다.
  - 검증 조건:
    - 결과: 호출이 페이지 재구성 텍스트 전체를 반환하고, 응답 보고 모델이 결정된 모델과 일치하며, 캐시
      읽기 토큰을 관찰할 수 있다.
    - 확인: 모킹 또는 주입 가능한 호출 경계로 스트리밍 누적·모델 반영·캐시 토큰 관찰을 단위 테스트로 확인한다.
  - 참조: SPEC §5.4, §5.5 / ANALYSIS §2, §5 D-e, D-f

- [ ] task-007: 동기 경로 오케스트레이션(선별→재구성→저장·기록, 부분 실패 보고)
  - 목적: 동기 경로를 실행하면 미번역·변경 페이지가 페이지마다 재구성·저장되고 그 즉시 증분 해시가 기록되며,
    실패 페이지는 전체를 중단시키지 않고 사유와 함께 보고된다.
  - 접근: 기존 대상 선별로 페이지 목록을 받아 페이지마다 동기 재구성을 호출하고, 성공분은 기존 저장+해시
    기록 함수(저장과 translated_hash 기록 동시)로 페이지별 즉시 저장·기록하며, 실패 페이지는 URL·사유를
    모아 보고한다. 출력 경로는 기존 경로 계산 함수로 구한다.
  - 검증 조건:
    - 결과: 성공 페이지는 출력 위치에 기존 규약대로 저장되고 해당 페이지의 translated_hash가 기록되며,
      일부 페이지 실패 시 성공분만 저장·기록되고 실패 페이지·사유가 보고된다.
    - 확인: 일부 성공·일부 실패가 섞인 입력에 대해 저장·기록·부분 실패 보고를 통합 테스트로 확인한다.
  - 참조: SPEC §5.1, §5.3, §5.7 / ANALYSIS §2, §5 D-g

## Section: Batch 경로

- [ ] task-008: Batch 제출과 페이지 식별자 바인딩
  - 목적: 전체 대상이 한 번에 Batch로 제출되고, 각 요청이 페이지 식별자를 실어 결과를 페이지로 역매핑할 수
    있는 제출 식별자를 확보한다.
  - 접근: SDK 소스로 배치 제출 심볼을 확인한 뒤, 페이지마다 공유 프리픽스+원문 요청을 구성하고 요청 식별자에
    페이지 식별자를 실어 일괄 제출하며 배치 식별자를 반환한다.
  - 검증 조건:
    - 결과: 대상 페이지 수만큼의 요청이 제출되고, 각 요청 식별자에 페이지 식별자가 실리며 배치 식별자가 확보된다.
    - 확인: 주입 가능한 호출 경계로 요청 구성·페이지 식별자 바인딩·제출을 단위 테스트로 확인한다.
  - 참조: SPEC §5.2 / ANALYSIS §2, §5 D-d

- [ ] task-009: Batch 폴링과 상태 관찰
  - 목적: 제출된 배치의 진행 상태를 폴링으로 추적해 제출·진행·완료 상태가 관찰되고, 완료 시 폴링이 종료된다.
  - 접근: SDK 소스로 배치 상태 조회 심볼을 확인한 뒤, 배치 식별자로 상태를 합리적 간격의 백오프로 폴링하며
    상태 전이를 관찰 가능하게 출력하고 완료 상태에서 종료한다.
  - 검증 조건:
    - 결과: 진행 중에는 폴링이 계속되며 상태가 관찰되고, 완료 상태가 되면 폴링이 종료된다.
    - 확인: 진행→완료로 변하는 모킹 상태 시퀀스에 대해 백오프 폴링·종료·상태 관찰을 단위 테스트로 확인한다.
  - 참조: SPEC §5.2 / ANALYSIS §2, §5 D-d

- [ ] task-010: Batch 결과 수집·역매핑과 저장·기록(부분 실패 보고)
  - 목적: 완료된 배치의 각 결과가 페이지로 역매핑되어 성공 페이지는 출력 위치에 저장·증분 기록되고, 실패
    페이지는 사유와 함께 보고된다.
  - 접근: SDK 소스로 배치 결과 수집 심볼을 확인한 뒤, 각 결과의 요청 식별자에서 페이지를 복원해 성공분은
    기존 저장+해시 기록 함수로 페이지별 저장·기록하고, 실패분은 페이지 URL·사유를 모아 보고한다.
  - 검증 조건:
    - 결과: 성공 페이지가 기존 규약대로 저장되고 translated_hash가 기록되며, 부분 실패 시 성공분만 저장·기록되고
      실패 페이지·사유가 보고된다.
    - 확인: 성공·실패가 섞인 모킹 결과 집합에 대해 역매핑·저장·기록·부분 실패 보고를 통합 테스트로 확인한다.
  - 참조: SPEC §5.2, §5.3, §5.7 / ANALYSIS §2, §5 D-d, D-g

- [ ] task-011: 새 서브커맨드와 모드 플래그(기본 Batch)
  - 목적: 사이트 식별자로 API 재구성 경로를 실행하는 서브커맨드가 추가되고, 기본은 Batch로, 플래그로 동기
    경로를 선택할 수 있으며, 처리 건수·실패 페이지·출력 경로가 보고된다.
  - 접근: 기존 plan/commit 코드를 손대지 않고 진입점 분기에 새 서브커맨드를 추가해, 사이트 로드→대상 선별→
    프리픽스 조립→경로 선택(기본 Batch, 모드 플래그로 동기)→저장·기록을 오케스트레이션하고 결과를 보고한다.
  - 검증 조건:
    - 결과: 모드 플래그 없는 실행은 Batch 경로로, 플래그 지정 시 동기 경로로 분기하고, 기존 plan/commit
      동작이 변하지 않으며, 처리 건수·실패·출력 경로가 출력된다.
    - 확인: 새 서브커맨드의 기본 Batch·플래그 동기 분기와 기존 plan/commit 무변경을 실행 또는 테스트로 확인한다.
  - 참조: SPEC §5.1, §5.2, §5.8 / ANALYSIS §1, §3, §5 D-a

- [ ] task-012: 증분·공존 e2e 확인
  - 목적: API 경로 실행 후 재실행하면 미번역·변경분만 대상이 되고(증분 유지), 기존 에이전트 기반 `/translate`
    절차가 변경 없이 그대로 사용 가능하다(두 경로 공존).
  - 접근: 임시 사이트로 API 경로 실행→재실행 시 처리 0건(무변경)→일부 원문 변경 후 변경분만 처리 흐름을
    확인하고, 기존 plan/commit·에이전트 절차의 입력·출력·매니페스트 규약이 동일함을 확인한다.
  - 검증 조건:
    - 결과: 무변경 재실행은 새로 처리되는 페이지가 없고, 일부 변경 시 그 페이지만 처리되며, 기존 plan/commit이
      같은 출력·매니페스트 규약으로 그대로 동작한다.
    - 확인: 실행→무변경 재실행(0건)→일부 변경 재실행(변경분만)과 기존 경로 무변경을 통합 테스트로 확인한다.
  - 참조: SPEC §5.3, §5.8 / ANALYSIS §1, §4, §5 D-g
