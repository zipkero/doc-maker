# translate-api-automation

번역의 "재구성(2단계)" 생성을 Anthropic Messages API 호출로 자동화하는 경로를 추가한다. 사람이 Claude Code를 직접
돌리지 않아도 수집된 원문이 한국어 문서로 재구성되며, 대량·반복 처리는 Batch + 프롬프트 캐싱으로, 단건·증분은 동기 호출로
처리한다. 기본 모델은 Sonnet 4.6다.

## 승인 전 확인
- 이 feature의 API 경로 도입이 부모 doc-localization SPEC §3의 "외부 번역 API 미호출" 제약을, 에이전트 경로 공존을
  전제로 대체·확장하는 것이 맞는지. 관련 본문: §1, §4

## 1. 범위
- 번역의 "재구성·번역 텍스트 생성"(translate 절차 2단계)을 Anthropic Messages API 호출로 수행하는 자동 경로를
  추가한다. 대상 선별(plan), 출력 저장, 매니페스트 기록(commit), 증분 판정 등 결정적 부분은 기존 그대로 유지한다.
- 실행 방식 두 가지를 둔다. 전체·대량 재번역은 Batch API(제출 → 폴링 → 결과 수집)로, 변경분 1~소수 페이지나
  디버깅은 동기 단건 호출로 처리한다. 두 경로 모두 기존 plan(대상 선별) 결과를 입력으로 받는다.
- 페이지 간 공유되는 프리픽스(재구성 규칙·용어집·스타일 기준)를 프롬프트 캐싱 대상으로 두어 페이지마다 재사용한다.
- 호출 모델은 사이트 설정으로 지정하며 기본값은 Sonnet 4.6다.
- 출력 경로·매니페스트·증분 동작은 기존 번역과 동일한 규약을 그대로 따른다(같은 output 위치, translated_hash 기록).
- 기존 에이전트 기반 `/translate` 절차는 변경 없이 그대로 두고, API 경로는 별도 진입점으로 공존시킨다.

## 2. 목표
- 사람이 에이전트를 직접 운전하지 않고도 수집 원문을 한국어 문서로 재구성·번역하는 자동 경로를 확보한다.
- 대량·반복·다중 사이트 번역을 사람 개입 없이 일괄 실행할 수 있게 한다.
- 공유 프리픽스 캐싱과 Batch로 대량 번역의 비용과 처리량을 개선한다.
- 단건·증분 변경은 동기 경로로 즉시 처리해 비동기 대기(최대 ~24h)를 피한다.

## 3. 제약
- Anthropic Messages API는 공식 Go SDK(anthropic-sdk-go)로 호출한다.
- 기본 호출 모델은 Sonnet 4.6(`claude-sonnet-4-6`)이며, 사이트 설정으로 다른 모델을 지정할 수 있다.
- API 키는 환경변수(`ANTHROPIC_API_KEY`)로 주입하며, 코드나 설정 파일·매니페스트에 저장하지 않는다.
- 재구성 품질 규칙(용어집 일관 적용, 상단 보일러플레이트 제거, 원문 병기 없음, 문서 끝 출처 한 줄, 기계친화 포맷의
  표·설명 재구성)은 기존 translate 절차와 동일하게 유지한다.
- 출력 경로 계산·매니페스트 기록·증분 판정(translated_hash 비교)은 기존 Go 결정적 부분을 재사용하며 동작이 바뀌지 않는다.
- 동기 경로에서 출력이 큰 페이지는 잘림을 피하도록 처리한다(충분한 출력 한도/스트리밍). Batch 경로는 비동기라 요청
  타임아웃 제약을 받지 않는다.
- 프롬프트 캐싱은 공유 프리픽스가 모델별 최소 캐시 토큰을 충족할 때만 적용되며, 미충족 시 캐시 없이 정상 동작한다.
- 외부 API 호출은 rate limit과 일시 오류에 대한 재시도를 고려한다.

## 4. 제외 범위
- 기존 에이전트 기반 `/translate` 절차의 제거·변경(별도 유지, 공존).
- 수집(collect) 단계의 변경.
- 한국어 외 다국어 출력(부모 feature와 동일하게 범위 밖).
- 번역 품질 자동 평가·회귀 검사.
- 페이지 유형별 자동 모델 분기(예: 산문은 저비용 모델, OpenAPI는 상위 모델). 이번 범위는 단일 기본 모델과 사이트
  설정 override까지로 한정한다.
- Batch·캐싱 외의 비용 최적화 기법.

## 5. 완료 조건
1. 동기 단건 경로를 실행하면, plan이 선별한 미번역·변경 페이지가 API 동기 호출로 재구성되어 그 사이트의 번역문 출력
   위치에 기존 경로 규약대로 저장된다(명령 실행 후 output 파일 생성과 처리 결과가 관찰된다).
2. Batch 경로를 실행하면, 대량·전체 대상이 Batch로 제출되고 폴링을 거쳐 완료 시 각 페이지 결과가 output 위치에
   저장된다. 제출·진행·완료 상태가 관찰된다.
3. 두 경로 모두 기존 plan 결과를 입력으로 하고, 저장 후 commit으로 매니페스트의 translated_hash가 기록되어, 재실행
   시 미번역분·변경분만 대상이 된다(증분 동작이 기존과 동일하게 관찰된다).
4. 모델을 지정하지 않으면 Sonnet 4.6로 호출되고, 사이트 설정으로 다른 모델을 지정하면 그 모델로 호출된다(응답에
   보고되는 모델이 설정값과 일치함이 관찰된다).
5. 페이지 간 공유 프리픽스가 프롬프트 캐싱으로 재사용되어, 응답 usage의 캐시 읽기 토큰으로 적중이 관찰된다(공유
   프리픽스가 최소 캐시 토큰 미만이면 캐시 없이 동작한다).
6. 재구성 결과가 기존 번역 규칙을 따른다: 용어집에 정의된 용어는 그 번역어로 일관 반영되고, 상단 보일러플레이트가
   제거되며, 원문 병기가 없고, 문서 끝에 출처 한 줄이 있으며, OpenAPI 등 기계친화 포맷은 표·설명 구조로 가공되어
   원문 그대로의 덩어리로 남지 않는다.
7. API 키가 없으면 명확한 오류로 중단하고(조용히 실패하지 않는다), 일부 페이지가 실패하면 실패한 페이지와 사유가
   보고되며 성공한 페이지의 결과는 저장된다.
8. 기존 에이전트 기반 `/translate` 절차가 변경 없이 그대로 사용 가능하다(두 경로 공존).
